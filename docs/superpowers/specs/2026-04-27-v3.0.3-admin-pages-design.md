# v3.0.3 ŌĆö Admin Pages Design Spec

**Branch:** `Kelvin-v3.0.3-AdminPages`
**Worktree:** `.worktrees/Kelvin-v3.0.3-AdminPages/`

**Project:** Swinburne Sarawak Library Self-Checkout System

## Goal

Build the three admin/staff backoffice pages that v3.0.2 testing surfaced as missing, plus a small set of supporting changes (CSV export, two database migrations to make schema reproducible).

## Scope

**In scope:**

1. **Admin Overdue list** ŌĆö `/dashboard/admin/overdue`. Actionable workboard for overdue loans. Filter, search, send reminder.
2. **Loan history** ŌĆö `/dashboard/staff/history`. Read-only report of all loans for admin + staff. Replaces the broken "Full history" link on admin/staff dashboards.
3. **Add new book** ŌĆö `/dashboard/admin/books/new`. Single-page form to create a Book + N Copies.
4. **CSV export** on (1) and (2).
5. **Housekeeping migrations**: capture `pg_trgm` extension + `search_books` RPC that v3.0.2 left in Supabase Dashboard but never wrote to a migration file.

**Out of scope (explicitly deferred):**

- Loan renewal flow (#4 from original brief) ŌĆö needs renewal policy fields first
- Fines management (#9) ŌĆö `loanPolicy.ts` has zero fines fields, requires its own design pass
- Tags/categories CRUD (#6) ŌĆö defer until pain-point identified after add-book lands
- Batch operations (#7), inventory check (#8) ŌĆö depend on #1ŌĆō#3 list pages first
- Email reminders ŌĆö v3.0.3 only writes in-app notifications
- xlsx export ŌĆö CSV is enough; no third-party packages

## Architecture

### Pattern reuse (mirror v3.0.2)

| New page | Template | Carry over |
|---|---|---|
| `/dashboard/admin/overdue` | `damage-reports` | server-fetch + URL-driven filters + AdminShell + client viewer |
| `/dashboard/staff/history` | `damage-reports` | same |
| `/dashboard/admin/books/new` | `admin/users` | client-fetch + per-feature server actions + ConfirmModal flow |

Server actions live in per-feature files (following v3.0.2's `damageActions.ts` convention):
- `app/dashboard/overdueActions.ts` ŌĆö sendReminder
- `app/dashboard/bookActions.ts` ŌĆö createBookWithCopies, lookupIsbnInDb, generateNextBarcodes

DB queries extend `app/lib/supabase/queries.ts`:
- `fetchOverdueLoans(filters)`
- `fetchAllLoansHistory(filters, page)`
- `findBookByIsbn(isbn)` (DB-only, no Open Library)
- `getNextAvailableBarcodes(count)`

CSV export piggybacks on the same page route via `?export=csv` searchParam ŌĆö server component returns a `Response` with CSV body instead of rendering React.

### Permissions

Existing `middleware.ts` already enforces:
- `/dashboard/admin/*` ŌåÆ admin only
- `/dashboard/staff/*` ŌåÆ staff + admin

New pages inherit. Each page also runs a `getDashboardSession()` + role check + `redirect` as the v3.0.2 double-guard pattern.

### Dashboard link updates

| File | Current link | New link |
|---|---|---|
| `app/ui/dashboard/admin/adminDashboard.tsx` "Full history" | `/dashboard/book/history` (404 trigger) | `/dashboard/staff/history` |
| `app/ui/dashboard/staff/staffDashboard.tsx` "Full history" | `/dashboard/book/history` (404 trigger) | `/dashboard/staff/history` |
| Admin dashboard | (no overdue entry yet) | add "View overdue loans" link ŌåÆ `/dashboard/admin/overdue` |
| Admin dashboard | (no add-book CTA yet) | add "Add new book" CTA ŌåÆ `/dashboard/admin/books/new` |

Student page `/dashboard/book/history` is **not** modified ŌĆö keeps its student-only `redirect` and personal-history view.

## Page 1 ŌĆö Admin Overdue (`/dashboard/admin/overdue`)

### Purpose

Admin-only actionable workboard. Lists every loan that is currently outstanding past its due date so admins can send reminders.

### Layout

`AdminShell` header: title `Overdue loans`, subtitle `Circulation ┬Ę Action required`, description `Loans past their due date. Send reminders or mark as contacted.`

**Filter row** (writes to URL, persists across reloads, shareable):
- Bucket pills: `All overdue` / `1ŌĆō7 days` / `8ŌĆō30 days` / `30+ days`
- Search input ŌĆö matches against borrower name, student ID, book title, copy barcode

**Table** (sorted by days overdue, descending):

| Column | Content |
|---|---|
| Book | Cover thumbnail (32├Ś46) + title + author |
| Borrower | Avatar + display_name + student_id + email |
| Borrowed | Date (e.g. `Mar 1, 2026`) |
| Due | Date in red (e.g. `Mar 31, 2026`) |
| Days overdue | Pill, red gradient by severity |
| Reminder status | Empty if never reminded; otherwise `Reminded 2h ago by Kelvin` |
| Action | `Send reminder` button |

**Top-right:** `Export CSV` link ŌåÆ `?export=csv` (preserves current filters).

**Empty state:** Centered card with green check icon, copy: `All loans are on time. Nice.`

### "Send reminder" behavior

A single click does all of:

1. Insert a row into `Notifications` for the borrower with body
   `Your loan of "Clean Code" is 27 days overdue. Please return it as soon as possible.`
2. Update the `Loans` row: `last_reminded_at = now()`, `last_reminded_by = staff_id`
3. Front-end optimistic update ŌĆö row re-renders to `Reminded just now by <staff>`
4. Toast `Reminder sent`

**Throttle:** If `last_reminded_at` is within the last 24 hours, the button is disabled with a tooltip `Reminded recently ŌĆö wait until tomorrow to resend`.

**No bulk-reminder action** in v3.0.3 ŌĆö too easy to spam students inadvertently. Defer to a later version.

**No email** in v3.0.3.

### Data scale

Single Sarawak campus, expected concurrent overdue loans: tens to low hundreds. Single fetch + client-side filtering is fine. No pagination. Re-evaluate at >500 rows.

### Server action

`app/dashboard/overdueActions.ts` ŌĆö `sendReminder(loanId: string)`:
- Verify staff/admin session
- Verify loan is currently overdue (defensive ŌĆö block on stale UI)
- Verify `last_reminded_at` is null or > 24h ago
- Insert Notification, update Loan in a single transaction
- `revalidatePath('/dashboard/admin/overdue')`
- Return `{ ok: true, lastRemindedAt }` for optimistic UI

### Query

`fetchOverdueLoans({ bucket?, search? }, sessionUser)`:
- `Loans.returned_at IS NULL AND due_at < now()`
- Joins to `Copies ŌåÆ Books`, `Users ŌåÆ UserProfile`
- Bucket = `1ŌĆō7 days` translates to `now() - due_at BETWEEN '1 day' AND '7 days'`
- Search across `Books.title`, `Books.author`, `Users.email`, `UserProfile.display_name`, `UserProfile.student_id`, `Copies.barcode`

## Page 2 ŌĆö Loan history (`/dashboard/staff/history`)

### Purpose

Read-only audit/report view for admin + staff. Sees every loan record across the library, all statuses. Differs from Overdue page in that it's archival (no buttons, just data).

### Layout

`AdminShell` header: title `Loan history`, subtitle `Circulation ┬Ę Archive`, description `All borrowing records. Filter, search, and export.`

**Filter row** (multi-faceted, all URL-backed):
- Status: `All` / `Active` / `Returned` / `Overdue` (default `All`)
- Time range: `Last 30 days` (default) / `Last 6 months` / `This semester` / `All time` / custom date pair
- Borrower search input
- Book search input
- Handler search input (optional)

**Summary line above table:**
> Showing 23 of 12,847 loans ┬Ę 8 active ┬Ę 12 returned ┬Ę 3 overdue

**Table** (default sort: `borrowed_at` desc):

| Column | Content |
|---|---|
| Book | Cover + title + author |
| Borrower | Display name + student_id |
| Borrowed | Date |
| Due | Date |
| Returned | Date or `ŌĆö` |
| Duration | `28d` (returned: actual; active: now() ŌłÆ borrowed_at) |
| Status | Color chip: `Active` (blue) / `Returned` (green) / `Overdue` (red) |
| Handler | Display name or `Self-checkout` |

**Pagination:** 50 rows/page, Previous/Next at bottom.

**Top-right:** `Export CSV` (current filters).

**Row expand (optional):** Click a row to reveal borrow notes, return notes, and a link to its damage report (if any).

**Default time range = `Last 30 days`** to avoid loading the full archive on first paint.

### Empty states

- 0 results across filters: card with `No loans match these filters.` + `Clear filters` button
- Page out of range: `No more results.` + `Back to first page`

### Explicitly NOT included

- ŌØī Edit / delete loan buttons ŌĆö loans are historical, mutate via DB tools
- ŌØī Send reminder button ŌĆö that's the Overdue page's job
- ŌØī Inline new-loan button ŌĆö checkout flow lives elsewhere

### Query

`fetchAllLoansHistory({ status, range, borrowerQ, bookQ, handlerQ }, page)`:
- Base query on `Loans`, joins same as `fetchRecentLoans`
- Status filter:
  - `Active`: `returned_at IS NULL AND due_at >= now()`
  - `Returned`: `returned_at IS NOT NULL`
  - `Overdue`: `returned_at IS NULL AND due_at < now()`
  - `All`: no filter
- Range filter on `borrowed_at`
- Returns `{ rows, totalCount }` for pagination summary

## Page 3 ŌĆö Add new book (`/dashboard/admin/books/new`)

### Purpose

Admin-only single-page form to catalog a brand-new title and create N copies. No external API calls ŌĆö all data entered manually.

### Layout (top to bottom)

**Section 0 ŌĆö AdminShell header:** title `Add new book`, subtitle `Catalog ┬Ę New title`, description `Create a new title and its physical copies.`

**Section 1 ŌĆö ISBN duplicate-check**

Single row, three controls:
- ISBN input (number-only, 10 or 13 digits)
- `Lookup` button
- `Scan ISBN` button (opens camera; reuses v3.0.2 `cameraScan` component)

Behavior:
- On lookup or successful scan, hit `findBookByIsbn(isbn)` against the **library's own database only**
  - **Match found:** show inline card `This book already exists: "Clean Code" by Robert C. Martin (3 copies)`. Two buttons:
    - `Go to that book` ŌåÆ redirect to the book's manage page
    - `Continue creating new entry` ŌåÆ dismiss card, focus first empty field (rare edge case for distinct editions sharing ISBN)
  - **No match:** small info note `ISBN not in library. Please fill in the details below.` and the ISBN value pre-fills the field below

**Section 2 ŌĆö Basic info**

Plain form fields:
- Title * (required, max 200)
- Author * (required, max 200; comma-separate multiple)
- ISBN (auto-filled from Section 1, editable)
- Publisher
- Publication year (number, 4 digits)
- Classification (e.g. `005.1 MAR`)
- Cover image URL (optional, paste-only; gradient placeholder if empty)

**Section 3 ŌĆö Category & tags**

- Category dropdown ŌĆö fixed 4 options: `Computer Science` / `Business` / `Art & Design` / `Engineering`
- Below dropdown: `Suggest from title` button ŌåÆ calls existing `/api/book/auto-category` (single-book mode), pre-selects the result; user can override
- Tags input ŌĆö chip-style. Type-and-enter to add. Below: `Suggest tags` button ŌåÆ calls existing `/api/book/auto-tag` for this single book

Both AI helpers are **optional** ŌĆö admin can skip them and select/type manually. Failures fall through silently (the existing API has heuristic fallback).

**Section 4 ŌĆö Copies**

- `Number of copies` input (min 1, max 20, default 1)
- Live preview block underneath:
  > **Will create barcodes:** `SWI-00213`, `SWI-00214`, `SWI-00215`

Preview computes by querying current max numeric suffix on `Copies.barcode WHERE barcode LIKE 'SWI-%'`, taking +1 as start.

No editable barcode input. Auto-generation only.

**Section 5 ŌĆö Submit**

Centered button: `Create book + 3 copies` (number tracks copies field).

On click ŌåÆ `ConfirmModal` (existing v3.0.2 component):
> You are about to create *Clean Code* with 3 copies (SWI-00213 to SWI-00215). Proceed?

`Confirm` runs server action `createBookWithCopies(...)`.

### After submit

- Success: green toast `Book created with 3 copies`, redirect to the new book's manage page (where admin can verify and possibly add more copies later)
- Failure: red toast `Failed to create book: <reason>`, form data preserved

### Server action

`app/dashboard/bookActions.ts` ŌĆö `createBookWithCopies(input)`:
- Verify admin session
- Validate input (title/author non-empty, ISBN format if provided, copies Ōłł [1, 20])
- In a single transaction:
  - Insert `Books` row
  - Compute next barcode range (lock + max query ŌĆö see "Barcode generation race")
  - Insert N `Copies` rows with status `available`
- `revalidatePath('/dashboard/admin/books/new')`, `revalidatePath('/dashboard/book/items')`
- Return `{ ok: true, bookId }` for redirect target

### Barcode generation race

If two admins submit simultaneously, both might compute the same "next" start barcode. Mitigation:
- Run barcode max-query inside a `SELECT ... FOR UPDATE` lock on a sentinel row (or wrap the whole insert in a serializable transaction)
- On unique-constraint violation (Copies.barcode is unique), retry once with refreshed start
- After 2 failed retries, return error `Could not generate barcode, please try again`

### Permissions

- Middleware: `/dashboard/admin/*` ŌåÆ admin only Ō£ģ
- Page header: re-check `getDashboardSession()` and redirect non-admins
- Server action: re-verify admin role before write

### Camera scan integration

Reuse `cameraScan` component. When opened from this page:
- On scan success ŌåÆ close modal, populate ISBN field, auto-fire `Lookup`
- On scan fail / cancel ŌåÆ just close modal, keep typing flow available
- On camera permission denied ŌåÆ toast `Camera not available ŌĆö please type ISBN manually`

## CSV export

Shared mechanism for Overdue and History pages.

### UX

- Top-right `Export CSV` link on each page
- Click ŌåÆ browser downloads a `.csv` file
- Filename: `<page>-YYYY-MM-DD.csv` (e.g. `overdue-2026-04-27.csv`, `loan-history-2026-04-27.csv`)
- Content respects current filters (not pagination ŌĆö exports the full match set)

### Implementation

Inside the existing server-component page, branch on `searchParams.export === 'csv'`:

```ts
if (params.export === 'csv') {
  const rows = await fetchOverdueLoans(filters);  // or fetchAllLoansHistory with no page limit
  const csv = '’╗┐' + buildCsv(rows);          // BOM for Excel UTF-8 detection
  return new Response(csv, {
    headers: {
      'Content-Type': 'text/csv; charset=utf-8',
      'Content-Disposition': `attachment; filename="overdue-${formatDate(now())}.csv"`,
    },
  });
}
```

Link in the UI is just an `<a href>` to the same path with `?export=csv&...current filters`. No new route, no new dependency.

### Columns

**Overdue CSV:**
Loan ID ┬Ę Title ┬Ę Author ┬Ę ISBN ┬Ę Copy barcode ┬Ę Borrower name ┬Ę Student ID ┬Ę Email ┬Ę Borrowed ┬Ę Due ┬Ę Days overdue ┬Ę Last reminded at ┬Ę Last reminded by

**History CSV:**
Loan ID ┬Ę Title ┬Ę Author ┬Ę ISBN ┬Ę Copy barcode ┬Ę Borrower name ┬Ę Student ID ┬Ę Email ┬Ę Borrowed ┬Ę Due ┬Ę Returned ┬Ę Duration days ┬Ę Status ┬Ę Handler

### CSV details

- UTF-8 with BOM (Excel default-opens correctly, no garbled text)
- Quoted fields when value contains `,` `"` or newline
- `"` inside fields escaped as `""`
- ISO date format `2026-04-27` (Excel auto-detects; Sheets too)
- One reusable helper: `lib/csv.ts` (small, self-contained)

### Out of scope

- ŌØī xlsx export (third-party dep)
- ŌØī Async/email delivery (data volume doesn't warrant it)
- ŌØī Progress indicator (synchronous download is fast enough)

## Error handling and edge cases

### Overdue page

| Situation | Behavior |
|---|---|
| Database fetch fails | `Failed to load overdue loans. Please try again.` + retry button |
| Reminder within 24h | Button disabled, tooltip `Reminded recently` |
| Loan returned by student during admin's session | Server action re-checks state; returns `{ ok: false, reason: 'already_returned' }`; toast surfaces; row removed from list on refresh |
| Two admins click reminder simultaneously | Both notifications go through (acceptable); `last_reminded_at` reflects whichever wrote second |

### History page

| Situation | Behavior |
|---|---|
| 0 results | Empty card + `Clear filters` |
| Custom range with end < start | Inline error `End date must be after start date`; submit blocked |
| Page number out of range | Redirect to page 1 |

### Add Books page

| Situation | Behavior |
|---|---|
| ISBN already in library | Inline card with "Go to that book" + "Continue anyway" |
| ISBN format invalid | Red helper text on field; lookup button disabled |
| Barcode generation fails after 2 retries | Toast `Could not generate barcode, please try again`; no partial writes |
| AI auto-category/auto-tag fails | Silent ŌĆö existing API already has heuristic fallback |
| Camera permission denied | Toast + revert to text-input mode |
| Network drop during submit | Toast `Failed to create book`; form values preserved |

### Global

| Situation | Behavior |
|---|---|
| Session expired | Middleware redirects to `/login?callbackUrl=...` (existing) |
| Wrong role tries to direct-navigate | Middleware redirects to role's home (existing) |

## Open implementation questions (for writing-plans phase)

These are settled enough to design but small details to nail when writing the implementation plan:

1. **Whose timezone is `due_at` compared against?** Server-side `now()` is UTC; "27 days overdue" should be calendar-day in Asia/Kuching. Implementation will use a small date helper to compute days overdue in MYT.
2. **Cover image URL validation** ŌĆö block obviously invalid URLs at submit time? Or accept anything and let the rendering fall back to gradient?
3. **CSV row cap** ŌĆö if a filter matches 50,000+ rows, do we still produce a single CSV? For Sarawak campus volume, this won't happen in v3.0.3 but worth a one-liner safety: cap at 10,000 rows in CSV output, truncation message at the top.
4. **`Books.isbn` UNIQUE constraint?** ŌĆö the "Continue creating new entry" path in the ISBN duplicate-check flow assumes two `Books` rows can share an ISBN (rare edition-clash case). During implementation, check the live schema: if `UNIQUE(isbn)` exists, either drop the constraint, or remove the "Continue anyway" option and force admins to use the existing book's manage page.

These don't block the spec; they are tactical decisions for `writing-plans` to lock down.
